# Architecture Decision Record 0004

## Title

Centralized Service Registry

---

## Status

Accepted

---

## Date

2026-06-20

---

# Context

The Hybrid Administration Platform is composed of many independent modules.

Examples include:

* Application Services
* Authentication
* Providers
* Caching
* Logging
* Telemetry
* Workflows

Without a centralized mechanism for resolving dependencies, modules would construct or import one another directly.

This creates:

* Tight coupling
* Circular dependencies
* Difficult testing
* Reduced extensibility

As the platform grows, direct module dependencies become increasingly difficult to maintain.

---

# Decision

HAP uses a centralized Service Registry as its dependency resolution mechanism.

Modules register services during platform initialization.

Consumers request services from the registry rather than creating implementations directly.

The bootstrap process is solely responsible for constructing the application.

---

# Responsibilities

The Service Registry is responsible for:

* Service registration
* Service resolution
* Lifetime management
* Dependency isolation
* Mock service substitution
* Platform composition

The Service Registry is **not** responsible for business logic.

---

# Provider Resolution

Providers are registered through the Service Registry.

Application services request providers through provider contracts.

They never instantiate provider implementations directly.

This allows:

* Mock providers
* Active Directory providers
* Graph providers
* Future providers

to be substituted without changing application logic.

---

# Rationale

## Loose Coupling

Application components depend upon contracts rather than implementations.

---

## Testability

Mock services can replace production services without modifying application code.

This significantly simplifies automated testing.

---

## Extensibility

Future providers and services integrate through registration rather than code modification.

---

## Composition

Application composition occurs once during startup.

After initialization, modules interact only through registered contracts.

---

## Consistency

Every subsystem follows the same dependency resolution model.

Developers never need to determine whether a component should be constructed manually.

---

# Consequences

Positive outcomes include:

* Reduced coupling
* Simplified testing
* Improved extensibility
* Cleaner module boundaries
* Consistent dependency management
* Easier platform composition

Trade-offs include:

* Slightly more startup complexity
* Additional abstraction layer
* Requirement for registration during bootstrap

These trade-offs are acceptable because they greatly improve long-term maintainability.

---

# Alternatives Considered

## Direct Module Imports

Modules import and construct one another directly.

Rejected because it creates tight coupling, circular dependencies, and difficult testing.

---

## Static Global Objects

Expose shared objects through global variables.

Rejected because global state is difficult to manage, difficult to test, and encourages hidden dependencies.

---

## Provider-Owned Dependency Management

Allow each provider to construct its own dependencies.

Rejected because it duplicates infrastructure, complicates testing, and violates separation of responsibilities.

---

# Outcome

The Service Registry becomes the composition root of the Hybrid Administration Platform.

Every subsystem—including providers, authentication, caching, logging, workflows, and future platform services—participates in a consistent dependency resolution model.

This architecture supports testing, extensibility, and long-term platform evolution while preventing direct coupling between independently developed modules.
