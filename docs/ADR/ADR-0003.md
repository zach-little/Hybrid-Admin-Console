# Architecture Decision Record 0003

## Title

Organization and Tenant Separation

---

## Status

Accepted

---

## Date

2026-06-20

---

# Context

Early versions of the Hybrid Administration Platform assumed that an organization corresponded to a single Microsoft tenant.

While this assumption simplifies implementation, it does not accurately represent many enterprise environments.

Organizations increasingly operate across multiple Microsoft cloud environments, including:

* Commercial
* GCC
* GCC High
* DoD

In some cases, a single organization simultaneously operates users, devices, workloads, and administrative staff across multiple Microsoft tenants and cloud environments.

Future platform growth also requires support for acquisitions, mergers, business units, and additional cloud providers.

A one-to-one relationship between organizations and tenants would unnecessarily restrict the platform and require future architectural redesign.

---

# Decision

Organizations and tenants are separate architectural concepts.

An Organization represents a customer or administrative boundary.

A Tenant represents a technology environment managed by the platform.

Organizations may contain zero or more tenant profiles.

Each tenant profile contains its own configuration and operational context.

Providers execute operations against Tenant Contexts rather than Organizations.

---

# Tenant Context

Each Tenant Context defines the operational environment for provider execution.

Examples include:

* Cloud Environment
* Tenant Identifier
* Authentication Configuration
* Provider Configuration
* Service Endpoints
* Provider Capabilities
* Default Policies

Tenant Contexts are resolved before provider execution.

Providers should not infer tenant information from organization configuration.

---

# Organization Responsibilities

Organizations define administrative information including:

* Organization Name
* Branding
* Default Configuration
* Available Providers
* Administrative Preferences
* Profile Metadata

Organizations do not contain provider-specific implementation details.

---

# Rationale

Separating organizations from tenants provides several architectural advantages.

## Multi-Cloud Support

Organizations can operate simultaneously within Commercial, GCC, GCC High, DoD, or future sovereign cloud environments.

---

## Scalability

Additional tenants may be introduced without changing platform architecture.

Future cloud providers automatically benefit from the existing tenant model.

---

## Flexibility

Organizations undergoing mergers, acquisitions, or cloud migrations can be represented without restructuring platform data.

---

## Provider Independence

Providers operate against explicit execution contexts rather than relying on implicit organization configuration.

This improves predictability and simplifies testing.

---

## Future Growth

The architecture naturally supports future expansion into additional cloud providers and technology platforms without requiring architectural redesign.

---

# Consequences

Positive outcomes include:

* True multi-tenant architecture
* True multi-cloud architecture
* Simplified provider implementation
* Improved scalability
* Improved configuration management
* Reduced customer-specific logic

Trade-offs include:

* Additional configuration complexity
* Explicit tenant selection requirements
* More sophisticated configuration management

These trade-offs are acceptable because they eliminate significant architectural limitations.

---

# Alternatives Considered

## One Organization Equals One Tenant

Treat every organization as a single Microsoft tenant.

Rejected because it cannot accurately model many enterprise environments and would require future architectural redesign.

---

## Cloud-Specific Organizations

Represent Commercial, GCC, GCC High, and DoD as separate organizations.

Rejected because cloud environments are technical deployment targets rather than administrative entities.

This approach would duplicate organization configuration and complicate management.

---

## Provider-Owned Tenant Configuration

Allow providers to manage tenant information independently.

Rejected because tenant identity is a platform concern rather than a provider concern.

Provider-specific tenant management would result in inconsistent behavior and duplicated configuration.

---

# Outcome

Organizations become administrative entities.

Tenants become execution environments.

Cloud environments become tenant attributes.

Providers execute against explicit Tenant Contexts supplied by the platform.

This architecture enables HAP to support organizations operating across multiple Microsoft tenants and cloud environments while preserving provider independence and maintaining a configuration-driven platform.
